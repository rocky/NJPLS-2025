Kinds of Containers

Where can a program come from?

  • a file in a file system,
  • interactive input; a stream from a socket
  • a piece of an "archive"-format file: Java classfile jar, Python wheel (zip), or machine-code library, etc.,
  • a string or AST data-structure created inside a program at runtime, e.g. HTML templating system for Web framework, an AI auto-learning system

Program source code comes from somewhere. It is useful for a debugger, profiler, or IDE to know where it came from. So what are some possibilities?

[DOWN fragment to file in filesystem] 8.1 ⟶ 8.1

As I said before, a file in the file system is the most common possibility.

A problem I see in many systems, is a weakness in being able to deal with positions when working with containers other than a file in a filesystem.

So let's go into other kinds of containers.

[DOWN fragment to stream] 8.1 ⟶ 8.1

Programs can come from interactive input; or more generally a stream from a socket. A weakness I see in compiler systems, is the inability to be able to show the source-text fragment in this situation. Here, input is often one line long. So it is useful to be able to give position information in finer granularity just a line.

[DOWN fragment to archive member] 8.1 ⟶ 8.1

In Java, a "class file", or jar, is in archive of a set of files. In Python, wheels are zip files.

[DOWN fragment reveal] 8.1 ⟶ 8.1

Another possibility is a string or AST data-structure created inside a program at runtime; for example an HTML templating system for Web framework, or self-modifying code in an auto-learning system.

Again, IDE's and debuggers are sometimes weak in being able to show source text when the container is of this type.
